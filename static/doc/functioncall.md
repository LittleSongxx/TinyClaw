# TinyClaw MCP / Skills / Function Calling Guide

TinyClaw currently uses a two-layer tool system:

- `MCP` is the transport and registration layer for external tools
- `Skills` are the higher-level task bundles that select tool subsets, prompts, memory policy, and execution hints

If you only remember one thing, remember this:

- `conf/mcp/mcp.json` controls what tools exist
- `skills/` controls the shipped local skill definitions
- `USE_TOOLS=true` is the recommended model-side tool switch for the current deployment

## Current Repository Defaults

By default, TinyClaw reads MCP config from:

```text
conf/mcp/mcp.json
```

You can override that path with:

```text
MCP_CONF_PATH
```

The current repository ships these MCP server entries in the default file:

- `playwright`
- `filesystem`
- `fetch`
- `time`
- `memory`
- `arxiv`
- `amap`
- `bocha-search`
- `github`

The current repository also ships these local skills under `skills/`:

- `general_research`
- `browser_operator`
- `workspace_operator`
- `github_operator`

## How Skills Are Built

At runtime, TinyClaw loads the skill catalog from three sources:

1. local skills from `skills/*/SKILL.md`
2. builtin fallback skills generated by code when a local skill with the same ID is absent
3. legacy proxy skills generated from the currently registered MCP tool set

That means:

- local `SKILL.md` files are the main project-owned source of truth
- builtin skills are safety nets, not the preferred customization surface
- legacy skills exist so every registered MCP server still has a compatibility path

The runtime also generates one catch-all fallback skill:

- `legacy_all_tools_proxy`

## Required Runtime Switches

Keep tool usage enabled in `deploy/docker/.env`:

```env
USE_TOOLS=true
```

If you use a custom MCP config file:

```bash
export MCP_CONF_PATH=/path/to/your/mcp_config.json
```

Then start TinyClaw normally:

```bash
./scripts/start.sh
```

## Secret-Based MCP Services

The default `conf/mcp/mcp.json` already includes several servers that depend on environment secrets:

```env
AMAP_MAPS_API_KEY=
BOCHA_API_KEY=
GITHUB_PERSONAL_ACCESS_TOKEN=
```

Without those values:

- the related MCP entries still exist in config
- the MCP or Skills pages may show warnings
- the corresponding tools or skills may be unavailable at runtime

## Docker Usage

With the current Docker Compose layout, the recommended setup is:

- keep using `conf/mcp/mcp.json` unless you truly need a separate file
- keep `USE_TOOLS=true` in `deploy/docker/.env`
- fill the required secrets for any enabled external MCP servers
- use the bundled `playwright-mcp` container through `http://playwright-mcp:8931/mcp`
- keep local skill files in `skills/`

## Admin Pages

The built-in admin panel exposes both layers:

- `#/mcp`
  inspect current MCP config, prepared templates, and availability warnings
- `#/skills`
  inspect the skill catalog, filter by source, validate the catalog, reload it, and view full skill detail

## Creating or Editing Local Skills

The supported local skill format is `SKILL.md` with YAML frontmatter plus required sections.

Each local skill must define:

- `id`
- `name`
- `description`
- `modes`
- `memory`
- `When to use`
- `When not to use`
- `Instructions`
- `Output contract`
- `Failure handling`

The runtime validates those fields when loading the catalog.

## Common Issues

### Logs show MCP connection failures

This usually means one of the following:

- the MCP service command is missing in the container
- the configured MCP URL is wrong
- the container cannot reach that address
- a required secret is still empty

### Skills page shows validation warnings

Check:

- `skills/*/SKILL.md` still has valid YAML frontmatter
- all required sections are present
- referenced MCP servers or tools are still available
- `conf/mcp/mcp.json` is readable and valid

### MCP is configured but the bot does not call tools

Check:

- `USE_TOOLS=true` is still enabled in the current maintained Docker stack
- the MCP services actually registered successfully
- the task prompt is suitable for tool invocation
- the relevant skill still resolves to usable tools
